## What This Project Is

Flick is a mobile-first web platform for collecting event photos and videos via QR code.
Hosts create a gallery in under 60 seconds and get a QR code. Guests scan it, upload photos
and videos directly from their phone browser — no app install, no signup required.
Hosts manage galleries, moderate uploads, and share a single link with everyone.

**Target users:** Indian weddings, college fests, birthday parties, corporate events, trips.
**Primary market:** India (WhatsApp sharing is a first-class feature).

## Database Schema (Convex)

### events
```
_id          auto
title        string
slug         string        unique, 8-char nanoid, lowercase alphanumeric
coverImage   string|null   UploadThing URL
eventDate    number        Unix ms
revealAt     number        Unix ms
hostId       string        Clerk userId — NEVER expose in public queries
createdAt    number        Date.now()
uploadCount  number        default 0
isPublic     boolean       default true
revealMode   "instant" | "after_event" | "next_day"
```

### uploads
```
_id          auto
eventId      Id<"events">
fileUrl      string
fileType     "image" | "video"
uploadedBy   string        required — guest-chosen username, stored from localStorage
uploadedAt   number        Date.now()
size         number        bytes
deviceInfo   string        user-agent, max 200 chars
isVisible    boolean       default true
```

### analyticsEvents
```
_id          auto
eventId      Id<"events">|null
eventType    "event_created"|"qr_opened"|"upload_started"|
             "upload_completed"|"gallery_opened"|"revisit"|"download_gallery"
timestamp    number
metadata     any
```

### Indexes
- events: `by_slug` (slug), `by_host` (hostId)
- uploads: `by_event` (eventId), `by_event_time` (eventId, uploadedAt)
- analyticsEvents: `by_event` (eventId), `by_type` (eventType)

## Convex Functions

### convex/events.ts
- `createEvent` — mutation, requires Clerk auth, generates slug with nanoid, retries up to 5 times on collision
- `getEventBySlug` — public query, never returns hostId
- `getEventsByHost` — protected query, requires userId match
- `updateUploadCount` — internalMutation, called from UploadThing callback

### convex/uploads.ts
- `addUpload` — mutation, called from UploadThing onUploadComplete
- `getUploadsByEvent` — public query, only returns isVisible=true records
- `deleteUpload` — mutation, requires host ownership check

### convex/analytics.ts
- `trackEvent` — mutation, fire-and-forget, never awaited in UI
- `getAnalyticsByEvent` — protected query, requires host ownership

**Security rule:** Every mutation that modifies event data must verify
`ctx.auth.getUserIdentity().subject === event.hostId`. Throw `ConvexError("Unauthorized")` if not.

## Authentication Rules

- ClerkProvider wraps the entire app in `src/app/layout.tsx`
- `middleware.ts` protects `/dashboard/*` only
- `/e/*` routes are always public — no auth middleware applied
- `getCurrentHost()` in `src/lib/auth.ts` — throws if no session, used in server components
- Guests never see a sign-in prompt
- Guests **must choose a username** before uploading — the name input on the event page is **required**, not optional
- Guest username is stored in `localStorage` under the key `flick_guest_name` so returning guests don't have to retype it
- Username is passed as `uploadedBy` to every upload made in that session
- No account is created — the username is display-only, stored only in uploads records

## UploadThing File Router

Two routes in `src/app/api/uploadthing/core.ts`:

```
eventCoverUploader
  - accepts: image/jpeg, image/png, image/webp
  - max: 4MB, single file
  - requires: Clerk auth (hosts only)

guestMediaUploader
  - accepts: image (jpg/png/webp/heic, 10MB) + video (mp4/mov/webm, 50MB)
  - max: 5 files per request
  - requires: NO auth
  - uploadedBy passed as input metadata from the client (required — validated server-side, reject if missing or empty)
  - onUploadComplete: calls Convex addUpload + updateUploadCount, passes uploadedBy from metadata
  - rate limit: 20 uploads per IP per hour per event (in-memory Map)
  - cap: reject if event uploadCount >= 500
```

## Analytics Events

Every key action is tracked in **both** PostHog and Convex `analyticsEvents`.

| Event | Trigger |
|---|---|
| `event_created` | After Convex createEvent succeeds |
| `qr_opened` | On `/e/[slug]` mount when `?src=qr` is in URL |
| `gallery_opened` | On `/e/[slug]` mount, always |
| `upload_started` | When UploadZone begins upload |
| `upload_completed` | In UploadThing onUploadComplete callback |
| `download_gallery` | When "Download All" is clicked |
| `revisit` | On gallery_opened if localStorage flag already set |

PostHog tracking is fire-and-forget. Convex tracking is fire-and-forget (never awaited in UI).

## Mobile-First Rules

Design for 375px viewport first, then enhance for desktop.

- All touch targets: minimum `min-h-11 min-w-11` (44×44px)
- UploadZone: "Tap to add photos" button on mobile, drag-drop zone on desktop
- Gallery: `columns-2` → `md:columns-3` → `xl:columns-4`
- Hero headline: `text-3xl md:text-6xl`
- Body text: never below `text-sm` for primary content
- MobileNav: `fixed bottom-0`, `flex md:hidden` — only on event pages
- Top Navbar: `hidden md:flex` nav links, hamburger on mobile

## Security Rules

1. Upload rate limit: 20 per IP per hour per event (in-memory Map in UploadThing middleware)
2. File type allowlist: `image/jpeg, image/png, image/webp, image/heic, video/mp4, video/quicktime, video/webm`
3. Input sanitisation: title max 100 chars, uploadedBy max 50 chars (required, reject if blank), deviceInfo max 200 chars
4. Upload cap: 500 per event — throw `ConvexError("Upload limit reached")`
5. All host mutations: verify `userId === event.hostId` before any write
6. Slug collision: retry up to 5 times before throwing
7. Guest username validation: `uploadedBy` must be a non-empty string after trim — block the upload UI until a name is entered

## Do Not Build (Post-MVP)

These are explicitly out of scope. Do not implement them even if it seems easy:

- Comments, likes, or reactions on photos
- Native iOS or Android apps
- AI-generated highlight reels or editing
- Social feed, discovery, or public gallery browsing
- Livestreaming
- Facial recognition or auto-tagging
- Paid tiers or Stripe integration
- Multi-host collaboration on a single event
